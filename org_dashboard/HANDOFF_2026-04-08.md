# Org Dashboard Handoff

## 1. 목적
- 이 문서는 `org_dashboard` 작업을 새 스레드에서 이어받기 위한 핸드오프입니다.
- 현재 운영 배포본과 로컬 테스트본이 분리되어 있습니다.
- 이번 저장 커밋은 **배포 목적이 아니라 작업 스냅샷 저장 목적**입니다.

## 2. 현재 상태 요약

### 운영 배포본
- 운영 페이지는 GitHub Pages `main` 기준으로 배포되어 있습니다.
- 최근 운영본에는 아래가 반영되어 있습니다.
  - 기존 조직구성 대시보드
  - 자리배치 탭
  - 관리자모드
  - 구글시트 연동
  - `CCMM11층` 추가

### 로컬 작업본
- 현재 로컬에는 운영본 위에 추가 실험 기능이 들어가 있습니다.
- 이번 로컬 작업의 핵심은 아래 2개입니다.
  1. `CCMM11층`을 포함한 확장된 테스트 대시보드
  2. `crop / 미니맵 / 12층 북측·남측 분리` 테스트 버전

## 3. 지금까지 구현된 주요 기능

### A. 기존 자리배치 기능
- 층별 좌석도 렌더링
- 이동 인원 강조
- 호버 시 원래 위치 정보 표시
- 관리자모드에서 좌석 선택 / 이동 대상 지정 / 이동 Y/N / 저장하기
- 구글시트 `seat_layout_latest` 읽기 및 저장
- 조직 레이어 ON/OFF

### B. CCMM11층 추가
- 엑셀 기반으로 `CCMM11F` 좌석도 생성
- 같은 seat layout 렌더러로 표시
- `ccmm_layout_latest` 시트 지원
- `seat_layout_latest + ccmm_layout_latest` 병합 로딩 가능
- CCMM 전용 예외 반영
  - `C12 최우석 / 인턴사원`
  - `C23 정서윤 / 인턴사원`
  - `C36 모션데스크`

### C. 조직 레이어
- `team_org_id_seat` 기준으로 조직 오버레이 생성
- 층별 라벨링 규칙 반영
  - 2층, 13층: 그룹 기준
  - 12층: 일부 그룹 기준 + 나머지 파트/센터 기준
  - CCMM: 그룹명 + 파트명 형태
- 집무실 내부의 조직장 좌석은 오버레이 제외

### D. 새 실험 기능
- `crop 확대 뷰`
- `12층 북측 / 남측 / 전체` 전환
- `미니맵 ON/OFF`

## 4. 지금 하고 있는 작업의 의미

현재 작업은 단순히 좌석을 더 그리는 것이 아니라,
**자리배치 화면을 실제 운영 가능한 도구로 바꾸는 단계**입니다.

핵심 방향:
- 구글시트에서 최신 좌석 상태를 불러옴
- 대시보드에서 시각적으로 이동/조직/좌석을 확인
- 관리자모드로 수정 후 다시 구글시트에 저장
- 필요하면 층별 확대 뷰와 미니맵으로 가독성 개선

즉 최종 목표는 아래입니다.
- `보고용 시각화`
- `운영용 수정도구`
- `구글시트 기반 최신 상태 관리`

## 9. 새 스레드에서 고정적으로 요청할 세팅

새 스레드에서 아래처럼 시작하면 이어받기 쉬움:

### 추천 시작 요청문
```md
이전 org_dashboard 작업 이어서 진행해줘.
먼저 HANDOFF_2026-04-08.md 를 읽고 현재 상태를 파악한 뒤,
로컬 테스트 파일 기준으로만 작업해줘. 운영본은 내가 확인 전까지 수정하지마.

작업 원칙:
- 설명보다 구현 우선
- 필요한 경우만 짧게 설명
- 로컬 테스트 URL 기준으로 검증
- 구글시트/Apps Script 수정은 통째로 교체 기준으로 안내
- 기존 디자인/조직 규칙을 최대한 유지
```

### 권장 요청 포인트
- `핸드오프 문서 먼저 읽기`
- `운영본 수정 금지, 로컬 테스트 우선`
- `설명보다 구현 우선`
- `Apps Script는 부분 수정 말고 전체 복붙 기준`

### 서브에이전트 역할 분리 추천
사용자가 명시적으로 허용할 때만 sub-agent를 쓰는 전제에서, 다음 역할 분리가 적절합니다.

1. `데이터/시트 에이전트`
- seat / assignments / Apps Script 응답 구조 점검
- 구글시트 컬럼, 시트명, payload 검증

2. `도면/렌더 에이전트`
- 좌석도 레이아웃
- crop, 미니맵, 조직 레이어
- CSS/렌더링 겹침 수정

3. `운영병합 에이전트`
- 테스트본 -> 운영본 병합 시 영향 범위 점검
- 배포 전 체크리스트

### 프롬프트 캐싱처럼 쓰는 팁
정식 캐시 기능보다, 새 스레드 첫 메시지에 아래 3개를 고정적으로 넣는 게 효과적입니다.
- `HANDOFF_2026-04-08.md 먼저 읽어`
- `로컬 테스트용으로만 작업`
- `운영본은 내가 확인 전까지 건드리지마`
